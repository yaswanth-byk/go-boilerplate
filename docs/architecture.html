Architecture

Understand how the monorepo is organized, how a request flows through the system, and which files are responsible for what.

Monorepo Structure

The project uses a Turborepo monorepo with two top-level directories:

📂 apps/

Deployable applications. Currently contains the Go backend. You can add frontends, admin panels, or other services here.

📦 packages/

Shared TypeScript libraries. Contains Zod schemas, OpenAPI generator, and email templates. Consumed by apps and each other.

Why a monorepo?

  • Shared types: Zod schemas define API contracts once, used by both frontend and backend
  • Atomic changes: Update a schema and its consumers in a single commit
  • Single CI pipeline: One repo to test, lint, and deploy everything
  • Turborepo caching: Only rebuild what changed — fast CI times

Backend Architecture

The Go backend follows a clean architecture pattern with clear separation of concerns. Dependencies flow inward — outer layers depend on inner layers, never the reverse.

┌──────────────────────────────────────────────────────────┐
│                     HTTP Request                         │
├────────────────────────┬─────────────────────────────────┤
│     Middleware Stack   │  Rate Limit → CORS → Auth →     │
│   (9 layers deep)      │  Request ID → Tracing → Logger  │
├────────────────────────┴─────────────────────────────────┤
│                      Router                              │
│              Maps URL → Handler                          │
├──────────────────────────────────────────────────────────┤
│                     Handlers                             │
│     Go generics for type-safe request/response           │
│     Automatic validation, logging, tracing               │
├──────────────────────────────────────────────────────────┤
│                     Services                             │
│     Business logic, orchestration, external APIs         │
├──────────────────────────────────────────────────────────┤
│                   Repositories                           │
│     Database queries (pgx), no business logic            │
├──────────────────────────────────────────────────────────┤
│                    Database                              │
│     PostgreSQL (pgxpool) + Redis (go-redis)              │
└──────────────────────────────────────────────────────────┘

Request Lifecycle

Here's exactly what happens when an HTTP request hits the server:

1

Rate Limiter

Checks if the client has exceeded 20 requests/second. If yes, returns 429 immediately without running any further middleware.

2

CORS

Adds CORS headers for allowed origins. Handles preflight OPTIONS requests automatically.

3

Security Headers

Adds X-Frame-Options, X-Content-Type-Options, X-XSS-Protection, and other security headers.

4

Request ID

Generates a UUID and attaches it to the request context. If the client sends an X-Request-ID header, that value is used instead. The ID is returned in the response headers.

5

New Relic Transaction

Starts a New Relic transaction. All database queries, Redis commands, and logs within this request are linked to this transaction.

6

Enhanced Tracing

Adds custom attributes (IP, user agent, request ID, user ID) to the New Relic transaction for better filtering in the dashboard.

7

Context Enrichment

Creates a per-request Zerolog logger pre-populated with request ID, method, path, IP, and trace IDs. This logger is stored in the Echo context so all downstream code can use it.

8

Request Logger

After the handler runs, logs the completed request with status code, latency, method, URI, and any error. Uses Info for 2xx, Warn for 4xx, Error for 5xx.

9

Panic Recovery

Catches any Go panics and converts them to 500 Internal Server Error responses instead of crashing the entire server.

10

Handler Execution

The router dispatches to the matched handler. The Handle generic function binds the request, validates it, calls your business logic, and returns the typed response.

Dependency Injection

Dependencies are wired manually in main.go (no framework). The Server struct acts as the dependency container:

// main.go — Dependency wiring
srv    := server.New(cfg, &log, loggerService)  // DB, Redis, Jobs
repos  := repository.NewRepositories(srv)        // Repos need DB
services := service.NewServices(srv, repos)      // Services need repos
handlers := handler.NewHandlers(srv, services)   // Handlers need services
router := router.NewRouter(srv, handlers, services) // Router needs handlers

This pattern is simple, explicit, and testable. In tests, you can substitute any dependency (e.g., use a test database instead of production).
