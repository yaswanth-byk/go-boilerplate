Server & Routing

The Server struct is the central dependency container. The router defines all endpoints and wires the middleware pipeline.

Files Involved

FilePurposeWhy It Exists
server.goDefines the Server struct that holds all shared dependencies (config, logger, DB, Redis, jobs). Manages HTTP server lifecycle (create, start, shutdown).You need a single place to hold all shared dependencies so they can be passed down to handlers, services, and repositories without globals.
router.goCreates the Echo router, attaches the full middleware pipeline, and registers route groups.Separating route registration from the server keeps the server focused on infrastructure and the router focused on HTTP concerns.
system.goRegisters "system" routes that don't require authentication: health check, static files, and API docs.System routes are separated from business routes so they can be reviewed and maintained independently.

The Server Struct

The Server struct acts as a dependency container. It holds every shared resource the application needs:

type Server struct {
    Config        *config.Config           // All configuration
    Logger        *zerolog.Logger           // Structured logger
    LoggerService *loggerPkg.LoggerService  // New Relic application
    DB            *database.Database        // PostgreSQL connection pool
    Redis         *redis.Client             // Redis client
    httpServer    *http.Server              // Go standard library HTTP server
    Job           *job.JobService           // Background job queue
}

Why not use a dependency injection framework?

Go favors explicit wiring over magic. The Server struct is created in main.go and passed to everything that needs it. This means:

  • You can see exactly what depends on what by reading the code
  • The compiler catches missing dependencies at build time
  • Tests can substitute any dependency easily

Server Lifecycle

1

server.New(cfg, logger, loggerService)

Creates the database pool (pgxpool with New Relic tracing), Redis client (with New Relic hooks), and starts the Asynq job server. If Redis is unavailable, a warning is logged but the server still starts.

2

srv.SetupHTTPServer(router)

Creates a net/http.Server with the Echo router as the handler. Applies read/write/idle timeouts from config.

3

srv.Start()

Calls ListenAndServe() on the configured port. This is run in a goroutine so the main goroutine can wait for shutdown signals.

4

srv.Shutdown(ctx)

Gracefully shuts down: stops accepting new connections → waits for in-flight requests (up to 30s) → closes database pool → stops job server.

Adding your own routes

// In router.go, after the system routes:
v1 := router.Group("/api/v1")

// Public routes
v1.GET("/products", handler.Handle(h.Product.Handler, h.Product.List, http.StatusOK, &ListProductsRequest{}))

// Protected routes (require authentication)
auth := v1.Group("", middlewares.Auth.RequireAuth)
auth.POST("/products", handler.Handle(h.Product.Handler, h.Product.Create, http.StatusCreated, &CreateProductRequest{}))
auth.PUT("/products/:id", handler.Handle(h.Product.Handler, h.Product.Update, http.StatusOK, &UpdateProductRequest{}))
