Testing

Integration tests with real PostgreSQL databases via Testcontainers. No mocks — every test runs against actual infrastructure.

Files Involved

FilePurposeWhy It Exists
container.goSpins up a PostgreSQL Docker container, creates a test database with a random name, applies all migrations, and returns a connection pool. Handles cleanup.Tests need a real database but shouldn't affect your development database. Each test gets an isolated, disposable PostgreSQL instance.
server.goCreates a test Server instance configured for testing (disabled observability, test credentials).Handlers and services need a *server.Server. This creates one pre-configured for the test environment.
helpers.goHigh-level SetupTest() that combines container + server setup in one call. Also provides utility functions like MustMarshalJSON(), ProjectRoot(), and Ptr[T]().Tests should be concise. SetupTest(t) gives you everything in one line.
assertions.goCustom assertion functions: AssertTimestampsValid(), AssertValidUUID(), AssertEqualExceptTime(), AssertStringContains().Common assertions for database-backed tests — checking UUIDs, timestamps, and struct equality while ignoring dynamic fields.
transaction.goTransaction helpers: WithTransaction() (commits) and WithRollbackTransaction() (always rolls back).Some tests need to write data and verify it (commit). Others need to write data, verify side effects, then undo everything (rollback). These helpers make both patterns easy.

How Testcontainers Works

Testcontainers uses the Docker API to create and manage containers programmatically from Go test code.

1

Docker starts PostgreSQL

A postgres:15-alpine container is started with a random database name (e.g., test_db_a1b2c3d4). The test waits for the "database system is ready" log message.

2

Connection pool is created

A pgxpool.Pool connects to the container. The test retries up to 5 times with 2-second delays (containers can take a moment to be ready).

3

Migrations are applied

All SQL migrations from internal/database/migrations/ are run against the test database. Your test starts with a fully migrated schema.

4

Test runs

Your test code runs against the real PostgreSQL instance. All queries, constraints, and triggers work exactly like production.

5

Cleanup

The connection pool is closed and the Docker container is terminated. Nothing persists between tests.

Writing a Test

Basic test structure

package myfeature_test

import (
    "testing"
    apptest "github.com/yaswanth-byk/go-boilerplate/internal/testing"
)

func TestCreateUser(t *testing.T) {
    // Setup: DB + server + cleanup — all in one line
    testDB, server, cleanup := apptest.SetupTest(t)
    defer cleanup()

    // testDB.Pool is a real PostgreSQL connection pool
    // server is a fully configured *server.Server

    // Your test logic here...
}

Using transaction helpers

func TestUserCreation(t *testing.T) {
    testDB, _, cleanup := apptest.SetupTest(t)
    defer cleanup()

    // Committed transaction — data persists for assertions
    err := apptest.WithTransaction(ctx, testDB, func(tx pgx.Tx) error {
        _, err := tx.Exec(ctx, `INSERT INTO users (name, email) VALUES ($1, $2)`, "John", "john@example.com")
        return err
    })
    require.NoError(t, err)

    // Verify the data persisted
    var count int
    testDB.Pool.QueryRow(ctx, `SELECT COUNT(*) FROM users`).Scan(&count)
    assert.Equal(t, 1, count)
}

func TestUserValidation(t *testing.T) {
    testDB, _, cleanup := apptest.SetupTest(t)
    defer cleanup()

    // Rollback transaction — data does NOT persist
    err := apptest.WithRollbackTransaction(ctx, testDB, func(tx pgx.Tx) error {
        // Insert and check a constraint violation
        _, err := tx.Exec(ctx, `INSERT INTO users (name, email) VALUES ($1, $2)`, "", "")
        return err  // Expect this to fail
    })
    assert.Error(t, err)  // Constraint violation
    // Table is empty — the insert was rolled back
}

Running Tests

# From apps/backend/

# Run all tests
task test

# Run tests with HTML coverage report
task test:cover

# Run a specific test
go test ./internal/handler/... -run TestHealthCheck -v

# Run tests in a specific package
go test ./internal/repository/... -v
⚠️ Docker required

Docker Desktop (or Docker Engine) must be running for tests that use Testcontainers. If Docker is not available, container-based tests will fail with a connection error.

💡 Test performance

Each test that calls SetupTest() starts a new PostgreSQL container. This takes 3-5 seconds. For test suites with many tests, consider sharing a single container across tests in the same package using TestMain().
