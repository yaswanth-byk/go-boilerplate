Observability

Structured logging with Zerolog and full-stack tracing with New Relic across HTTP, database, Redis, and application logs.

Files Involved

FilePurposeWhy It Exists
logger/logger.goCreates and configures the Zerolog logger and New Relic application. Handles environment-aware formatting and New Relic log forwarding.You need a single place to set up logging and observability for the entire app. This file does both.
config/observability.goDefines all observability configuration: logging levels/format, New Relic settings, health check intervals.Keeps observability settings organized and provides sensible defaults.
middleware/tracing.goEnriches New Relic transactions with request context (IP, user ID, request ID).Standard New Relic middleware only captures method/route/status. This adds the custom data you actually filter by.
middleware/context.goCreates per-request loggers with all context fields pre-populated.Every log line from a request automatically includes trace context — no manual work needed.

Logging with Zerolog

Zerolog is a high-performance, zero-allocation JSON logger for Go.

Environment-aware output

EnvironmentFormatExample Output
local / development Pretty console with colors and timestamps 
10:30:00 INF handling request handler=HealthHandler route=/status
production Structured JSON 
{"level":"info","handler":"HealthHandler","route":"/status","time":"2026-03-06T10:30:00Z"}

Using the logger in your code

The logger is available from the Echo context (enriched per-request) or directly from the server:

// From a handler — uses the per-request enriched logger
func (h *UserHandler) Create(c echo.Context, req *CreateReq) (*UserRes, error) {
    logger := c.Get("logger").(*zerolog.Logger)
    logger.Info().Str("email", req.Email).Msg("creating user")
    // Output: INF creating user email=john@example.com request_id=abc-123 ...
}

// From a service — uses the server-level logger
func (s *UserService) ProcessBatch() {
    s.server.Logger.Info().Int("count", 100).Msg("processing batch")
}

New Relic APM

New Relic provides application performance monitoring. The boilerplate has deep integrations across every layer:

🌐 HTTP Transactions

Every HTTP request is captured as a New Relic transaction via nrecho-v4. Includes method, route, status, response time, and custom attributes (IP, user ID, request ID).

🗄️ Database Queries

Every SQL query appears as a database segment via nrpgx5. Shows query text, duration, and which transaction triggered it.

🔴 Redis Commands

Every Redis command is traced via nrredis-v9. Shows command name, key, and duration.

📝 Log Forwarding

All Zerolog output is forwarded to New Relic Logs via zerologWriter. Logs are automatically linked to the transaction they belong to.

🐛 Error Tracing

Errors are reported to New Relic with full stack traces via nrpkgerrors. Compatible with pkg/errors wrapped errors.

📊 Custom Events

Custom events like RateLimitExceeded and HealthCheckFailed are sent to New Relic for alerting.

Disabling New Relic

Simply leave BOILERPLATE_OBSERVABILITY.NEW_RELIC.LICENSE_KEY empty in your .env. All New Relic integrations become no-ops — no code changes needed, no performance overhead.

The LoggerService Struct

type LoggerService struct {
    NRApp *newrelic.Application  // New Relic application instance
}

This struct wraps the New Relic application. It's passed to the server so database, Redis, and middleware can access it for instrumentation. If New Relic is disabled, NRApp is nil, and all integrations check for nil before using it.
