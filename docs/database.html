Database

PostgreSQL connection pooling with pgx v5, embedded SQL migrations with tern, and integrated query tracing.

Files Involved

FilePurposeWhy It Exists
database.goCreates the PostgreSQL connection pool (pgxpool.Pool), builds the DSN, configures tracing, and manages pool settings.Centralizes all database connection logic. Every part of the app that needs the database gets it from this single pool.
migrator.goRuns SQL migrations using tern. Migrations are embedded in the Go binary via go:embed.Ensures the database schema is always in sync with the application code. No separate migration tool needed in production.
migrations/*.sqlIndividual SQL migration files. Each file has an "up" section and a "down" section separated by a comment marker.Track schema changes over time with version control.

Connection Pool

The database layer uses pgx v5 with pgxpool for connection pooling. A connection pool maintains multiple open database connections and reuses them across requests, avoiding the overhead of opening a new connection for every query.

How the DSN is built

// Format: postgres://user:password@host:port/database?sslmode=X
dsn := fmt.Sprintf("postgres://%s:%s@%s:%d/%s?sslmode=%s",
    cfg.Database.User,
    url.QueryEscape(cfg.Database.Password),  // Handles special chars
    cfg.Database.Host,
    cfg.Database.Port,
    cfg.Database.Name,
    cfg.Database.SSLMode,
)
💡 Note

The password is URL-encoded with url.QueryEscape(). This means special characters like @, #, % in passwords work correctly without breaking the DSN URL.

Pool settings

SettingDefaultWhat It Does
MaxConns25Maximum connections the pool will open simultaneously. Set this based on your database's max_connections.
MinConns25Minimum idle connections to keep open. Pre-warms the pool so requests don't wait for connection creation.
MaxConnLifetime5 minMaximum time a connection stays open before being recycled. Prevents stale connections.
MaxConnIdleTime5 minMaximum time an idle connection sits unused before being closed.

Query Tracing (Multi-Tracer)

The connection pool is configured with a multi-tracer that pipes query information to multiple destinations:

🟢 New Relic Tracer

Every SQL query appears as a database segment in New Relic. You can see query text, duration, and which transaction triggered it. Only active when a New Relic license key is configured.

📝 Zerolog Tracer

In local environment only, SQL queries are logged to the console via pgx-zerolog. Useful for debugging — you can see every query that runs.

Using the Database in Code

The connection pool is available via server.DB.Pool:

// In a repository:
func (r *UserRepo) GetByID(ctx context.Context, id uuid.UUID) (*model.User, error) {
    var user model.User
    err := r.server.DB.Pool.QueryRow(ctx,
        `SELECT id, name, email, created_at, updated_at
         FROM users WHERE id = $1`, id,
    ).Scan(&user.ID, &user.Name, &user.Email, &user.CreatedAt, &user.UpdatedAt)

    if err != nil {
        return nil, sqlerr.HandleError(err)  // Auto-maps to HTTP errors
    }
    return &user, nil
}

Migrations

How migrations work

Migrations use tern (by the same author as pgx). Key features:

  • Embedded in the binary: Migration SQL files are compiled into the Go binary using //go:embed migrations/*.sql. This means you don't need to ship migration files separately in production.
  • Versioned: Each migration has a number prefix (001_, 002_, etc.). Tern tracks which migrations have been applied in a schema_version table.
  • Up/Down: Each file contains both "up" and "down" SQL separated by ---- create above / drop below ----.

Migration file format

-- 002_create_users.sql

-- Write your migrate up statements here
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name TEXT NOT NULL,
    email TEXT UNIQUE NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()
);

---- create above / drop below ----

-- Write your migrate down statements here
DROP TABLE IF EXISTS users;

Creating a new migration

cd apps/backend
task migrations:new name=create_users

This creates a numbered SQL file in internal/database/migrations/.

Running migrations

# Apply all pending migrations
task migrations:up
⚠️ Auto-migration behavior

Migrations run automatically on startup in all environments except local. In local development, you must run task migrations:up manually. This prevents accidental schema changes during development.
