TypeScript Packages

Three TypeScript packages in the monorepo generate API contracts, OpenAPI specs, and email templates consumed by the Go backend.

Why TypeScript in a Go Project?

These packages exist because certain tools in the TypeScript ecosystem (Zod for schema validation, ts-rest for OpenAPI generation, React Email for templates) are more mature and ergonomic than their Go equivalents. They generate static artifacts (JSON specs, HTML templates) that the Go backend consumes at runtime — there's no TypeScript running in production.

Monorepo Tooling

🔧 Turborepo

Orchestrates tasks (build, dev, lint, test) across all packages. Caches build outputs so only changed packages are rebuilt. Configured in turbo.json.

🐰 Bun

JavaScript runtime and package manager. Much faster than npm/yarn for installs and script execution. Used via bun install, bun run dev, etc.

Root-level commands

# Run from the project root
bun run build          # Build all packages
bun run dev            # Start all packages in dev mode (watch)
bun run lint           # Lint all packages
bun run lint:fix       # Lint with auto-fix
bun run typecheck      # Type-check all packages
bun run test           # Run all tests
bun run clean          # Clean all build artifacts

packages/zod — Zod Schemas

FilePurposeWhy It Exists
src/health.tsDefines Zod schemas for the health check response. Uses extendApi() from @anatine/zod-openapi to add OpenAPI metadata (descriptions, examples).These schemas serve as the single source of truth for API contracts. They can be used for both runtime validation and static type generation.
src/utils.tsUtility Zod schemas reused across multiple endpoints (e.g., UUID schema, paginated list schema, error response schema).Shared schemas avoid duplication and ensure consistency across all API endpoints.
src/index.tsBarrel file that re-exports all schemas from a single entry point.Consumers import from @repo/zod instead of individual files.
package.jsonPackage config, declares this as @repo/zod in the monorepo.Turborepo uses this to understand package dependencies and build order.
tsconfig.jsonTypeScript config using NodeNext module resolution.Ensures consistent TS behavior across the monorepo.

How Zod schemas work

// packages/zod/src/health.ts
import { z } from "zod";
import { extendApi } from "@anatine/zod-openapi";

export const HealthCheckResponseSchema = extendApi(
  z.object({
    status: z.string().describe("Overall health status"),
    timestamp: z.string().describe("ISO 8601 timestamp"),
    environment: z.string().describe("Current environment"),
    checks: z.object({
      database: z.object({
        status: z.string(),
        response_time: z.string(),
      }),
      redis: z.object({
        status: z.string(),
        response_time: z.string(),
      }),
    }),
  }),
  { description: "Health check response" }
);

packages/openai — OpenAPI Generation

FilePurposeWhy It Exists
src/index.tsDefines the OpenAPI contract using ts-rest. Maps HTTP endpoints to Zod schemas, sets up security schemes (Bearer JWT, API key).The contract is the bridge between TypeScript schemas and the OpenAPI spec. It says "GET /status returns HealthCheckResponseSchema".
src/gen.tsScript that generates the openapi.json file from the ts-rest contract. Outputs both openapi.json (backend) and openai.json (copy).Converts the TypeScript contract to an OpenAPI 3.0.2 JSON spec that the Go backend serves at /docs.
src/contracts/health.tsDefines the health check endpoint contract using ts-rest.Each feature area has its own contract file for organization.
openai.jsonThe generated OpenAPI spec file.This is the output artifact. Copied to apps/backend/static/openapi.json for serving.

Regenerating the OpenAPI spec

cd packages/openai
bun run build      # or: bun gen

This runs src/gen.ts, which reads the ts-rest contract, converts it to OpenAPI 3.0.2, and writes the JSON file.

packages/emails — Email Templates

FilePurposeWhy It Exists
src/templates/welcome.tsxWelcome email template built with React Email components.React Email lets you design emails with JSX instead of raw HTML. The output is compiled to HTML and placed in the Go backend's template/emails/ directory.
package.jsonPackage config with React Email dependencies.Declares @react-email/components as a dependency for building templates.

The email workflow

  1. Design emails in packages/emails/src/templates/*.tsx using JSX
  2. Preview them with React Email's dev server: cd packages/emails && bun run dev
  3. Build/export the HTML: the compiled HTML goes to apps/backend/template/emails/
  4. The Go backend reads these HTML files with html/template and sends via Resend
