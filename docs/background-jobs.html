Background Jobs

Redis-backed asynchronous task processing with Asynq. Run heavy or time-consuming operations outside the HTTP request cycle.

Files Involved

FilePurposeWhy It Exists
job.goCreates the Asynq client (for enqueueing) and server (for processing). Configures concurrency and priority queues. Manages lifecycle (start/stop).Email sending, report generation, and other slow operations shouldn't block HTTP responses. This file sets up the infrastructure to process them asynchronously.
email_tasks.goDefines task types (constants), payload structs, and task constructor functions for email-related jobs.Each job type needs a unique name, a serializable payload, and settings like retry count and timeout. This file is the registry.
handlers.goInitializes shared dependencies (like the email client) and contains the handler functions that process tasks.When a task is dequeued, Asynq calls the handler registered for that task type. This file maps task types to their processing logic.

What is Asynq?

Asynq is a Go library for Redis-backed task queues. It works like this:

  1. Producer: Your HTTP handler creates a task and enqueues it to Redis
  2. Redis: Stores the task in a queue until a worker picks it up
  3. Consumer: The Asynq server (running in a goroutine) dequeues tasks and calls the registered handler

Priority Queues

Three queues with different weights control which tasks get processed first:

QueueWeightUse Case
critical6Password resets, verification emails, payment confirmations
default3Welcome emails, notification emails, standard tasks
low1Analytics processing, report generation, cleanup tasks

Weights determine processing ratio. With 10 workers, roughly 6 will pull from critical, 3 from default, and 1 from low at any given time.

Enqueueing a Job

// In a service or handler:
task, err := job.NewWelcomeEmailTask("user@example.com", "John")
if err != nil {
    return err
}

// Enqueue the task — it will be processed asynchronously
info, err := server.Job.Client.Enqueue(task)
if err != nil {
    logger.Error().Err(err).Msg("failed to enqueue welcome email")
    return err
}
logger.Info().Str("task_id", info.ID).Msg("welcome email task enqueued")

Adding a New Job Type

Step 1: Define the task type and payload

// email_tasks.go (or create a new file for non-email tasks)

const TaskInvoiceGeneration = "invoice:generate"

type InvoicePayload struct {
    OrderID   string `json:"order_id"`
    UserEmail string `json:"user_email"`
}

func NewInvoiceTask(orderID, email string) (*asynq.Task, error) {
    payload, err := json.Marshal(InvoicePayload{
        OrderID:   orderID,
        UserEmail: email,
    })
    if err != nil {
        return nil, err
    }

    return asynq.NewTask(TaskInvoiceGeneration, payload,
        asynq.MaxRetry(3),               // Retry up to 3 times on failure
        asynq.Queue("default"),           // Use the default priority queue
        asynq.Timeout(60*time.Second),    // Timeout after 60 seconds
    ), nil
}

Step 2: Write the handler

// handlers.go

func (j *JobService) handleInvoiceTask(ctx context.Context, t *asynq.Task) error {
    var p InvoicePayload
    if err := json.Unmarshal(t.Payload(), &p); err != nil {
        return fmt.Errorf("failed to unmarshal invoice payload: %w", err)
    }

    j.logger.Info().Str("order_id", p.OrderID).Msg("generating invoice")

    // Your invoice generation logic here...

    return nil  // Return nil on success, error to trigger retry
}

Step 3: Register the handler

// job.go — in the Start() method

func (j *JobService) Start() error {
    mux := asynq.NewServeMux()
    mux.HandleFunc(TaskWelcome, j.handleWelcomeEmailTask)
    mux.HandleFunc(TaskInvoiceGeneration, j.handleInvoiceTask)  // Add this

    j.logger.Info().Msg("Starting background job server")
    return j.server.Start(mux)
}

Task Settings

OptionDefault (Welcome Email)Description
MaxRetry3Number of times to retry on failure. Uses exponential backoff.
QueuedefaultWhich priority queue to use (critical, default, or low).
Timeout30 secondsMaximum time the handler has to complete. Prevents hung tasks.
⚠️ Redis required

The job queue requires a running Redis instance. Without Redis, task enqueueing will fail. Make sure BOILERPLATE_REDIS.ADDRESS is set correctly in your .env.
