Error Handling

A three-layer error system that gives your API users clear, consistent, actionable error messages — including automatic mapping of database constraint violations to user-friendly HTTP responses.

Files Involved

FilePurposeWhy It Exists
errs/http.goDefines the HTTPError struct and constructor functions (NewBadRequestError, NewNotFoundError, etc.).Every API error should have a consistent JSON shape. This file defines that shape and provides helpers to create errors.
errs/type.goDefines error code constants (BadRequest, Unauthorized, etc.) and the ErrorAction struct for client-side redirects.Enumerates all possible error types so the frontend can switch on the error code.
sqlerr/error.goDefines PostgreSQL error codes, severity levels, and mapping functions.Maps low-level PostgreSQL error codes (like 23505) to semantic categories (like UniqueViolation).
sqlerr/handler.goThe HandleError() function — the main entry point for converting database errors to HTTP errors.You call sqlerr.HandleError(err) in your repository and get back a user-friendly HTTP error automatically.
middleware/global.goGlobal error handler that catches all errors (including Echo errors and panics).Last line of defense — ensures every error returns a consistent JSON format.

Error Response Format

Every error returns this JSON shape:

{
  "code": "BAD_REQUEST",
  "message": "A User with this Email already exists",
  "status": 400,
  "override": true,
  "errors": [
    { "field": "email", "error": "is required" }
  ],
  "action": {
    "type": "redirect",
    "message": "Session expired",
    "value": "/login"
  }
}
FieldTypeDescription
codestringMachine-readable error code (e.g., BAD_REQUEST, NOT_FOUND, USER_ALREADY_EXISTS)
messagestringHuman-readable error message for display
statusintHTTP status code (400, 401, 403, 404, 500)
overrideboolIf true, the frontend should display this message directly. If false, the frontend should use its own default message for this error code.
errorsarrayField-level validation errors (only for 400 errors with validation failures)
actionobjectOptional. Instructs the frontend to take an action (e.g., redirect to login).

Creating Errors

// 400 Bad Request
errs.NewBadRequestError("Invalid input", true, nil, nil, nil)

// 400 with custom error code
code := "USER_ALREADY_EXISTS"
errs.NewBadRequestError("User already exists", true, &code, nil, nil)

// 400 with field-level validation errors
fields := []errs.FieldError{
    {Field: "email", Error: "is required"},
    {Field: "name",  Error: "must be at least 2 characters"},
}
errs.NewBadRequestError("Validation failed", true, nil, fields, nil)

// 401 Unauthorized
errs.NewUnauthorizedError("Invalid token", false)

// 403 Forbidden
errs.NewForbiddenError("Insufficient permissions", true)

// 404 Not Found
errs.NewNotFoundError("User not found", true, nil)

// 500 Internal Server Error (generic, hides details)
errs.NewInternalServerError()

Automatic SQL Error Mapping

This is one of the most powerful features of the boilerplate. The sqlerr.HandleError() function automatically converts PostgreSQL constraint errors into user-friendly HTTP errors.

How it works

When a database query fails with a constraint violation, PostgreSQL returns a structured error with the table name, column name, and constraint name. The sqlerr package reads these fields and generates appropriate error messages.

PostgreSQL Error CodeHTTP StatusGenerated Message Example
23505 — Unique violation400"A User with this Email already exists"
23503 — Foreign key violation400"The referenced Organization does not exist"
23502 — Not null violation400"The Name is required"
23514 — Check violation400"The Age value does not meet required conditions"
No rows found404"User not found"
All other errors500"Internal Server Error"

Message generation logic

  • Table names are singularized (e.g., "users" → "User") and title-cased
  • Column names are humanized (e.g., "first_name" → "First Name")
  • Foreign key columns ending in _id are converted to entity names (e.g., "organization_id" → "Organization")
  • Error codes are generated from table + action (e.g., USER_ALREADY_EXISTS, ORGANIZATION_NOT_FOUND)

Usage in repositories

func (r *UserRepo) Create(ctx context.Context, name, email string) (*model.User, error) {
    var user model.User
    err := r.db.Pool.QueryRow(ctx,
        `INSERT INTO users (name, email) VALUES ($1, $2) RETURNING *`,
        name, email,
    ).Scan(&user.ID, &user.Name, &user.Email, &user.CreatedAt, &user.UpdatedAt)

    if err != nil {
        return nil, sqlerr.HandleError(err)
        // If email is duplicate → "A User with this Email already exists" (400)
        // If name is null     → "The Name is required" (400, with field errors)
        // If no rows          → "User not found" (404)
    }
    return &user, nil
}
💡 Key Insight

You don't need to manually check for constraint violations. Name your database constraints consistently (e.g., unique_users_email, fk_orders_user_id), and sqlerr will generate clear error messages automatically.
